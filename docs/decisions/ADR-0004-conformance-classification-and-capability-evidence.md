# ADR-0004: Classify conformance results and keep capability evidence deterministic

Status: Accepted

Date: 2026-07-18

Decision owners: Bootstrap maintainers

Decision record: [Bootstrap issue #58](https://github.com/OMT-Global/bootstrap/issues/58)

## Context

The first conformance report schema exposed stable rule IDs, severities, evidence, remediation, deterministic ordering, and blocking exit codes. It could not distinguish a control that was misconfigured from one unavailable on the current GitHub plan, and it did not identify results governed by an approved waiver. It also relied on ownership drift for many generated artifacts and did not independently inspect every checked-in workflow reference.

Remote GitHub capability state cannot be inferred reliably from repository files. Querying GitHub inside every conformance run would make otherwise reproducible evaluation depend on network, authentication, and the caller's privileges.

## Decision

Bootstrap conformance schema v2 adds a required classification to every result:

- `conformant` for present and valid controls;
- `misconfigured` for available controls that are absent or invalid;
- `unsupported` for controls unavailable on the observed GitHub plan;
- `waived` only for current typed manifest exceptions;
- `unverified` when local evidence cannot establish remote state.

The engine will also:

1. Check that every selected managed artifact exists independently of ownership drift.
2. Inspect every checked-in YAML workflow, including unmanaged workflows, for immutable action and reusable-workflow references.
3. Require readable release metadata for third-party action pins while continuing to require immutable references for first-party control-plane workflows.
4. Accept an optional versioned GitHub capability snapshot containing only `supported`, `unsupported`, or `misconfigured` observations.
5. Keep capability capture outside conformance evaluation so a caller can gather it with appropriate authorization and then evaluate it deterministically and offline.
6. Reject multiline, oversized, or credential-like capability evidence before it can enter human or retained JSON output.
7. Derive `waived` only from validated manifest exceptions; an observation file cannot create or approve a waiver.

## Consequences

- Consumers of conformance JSON must migrate from schema v1 to v2 and handle the required `classification` field.
- Blocking findings continue to return exit code `1`; warnings remain non-blocking and distinct.
- An unsupported GitHub feature is visible as a warning with remediation instead of being confused with repository misconfiguration.
- Capability snapshots are evidence inputs, not authority to change GitHub settings or approve exceptions.
- Repositories with mutable workflow references become nonconformant even when those workflows are not generated by Bootstrap.

## Alternatives considered

### Infer unsupported features from HTTP status text during conformance

Rejected because error strings vary by endpoint and plan, and the result would depend on the caller's token privileges and live network state.

### Treat unsupported controls as ordinary warnings without classification

Rejected because automation and maintainers could not distinguish plan limitations from correctable configuration drift.

### Allow snapshots to declare waivers

Rejected because a transient evidence file is not an approval record. Waivers remain typed, scoped, reviewable manifest exceptions.

## Rollout

1. Publish schema v2 with required-artifact, workflow-pin, and capability-classification coverage.
2. Update callers to branch on `classification` as well as `severity`.
3. Add security, provenance, and release capability observations as issues #60, #61, and #64 project their full controls.

## Security and privacy

Capability evidence is schema-bounded, single-line, and rejected when it resembles a credential. Workflow-derived findings redact credential-like literals and strip control characters before reporting. Snapshot capture must use the least GitHub privilege needed for the observed controls and should remain outside the repository when it contains non-public operational detail.

## Revisit conditions

Revisit if GitHub provides a stable capability API that can be captured without privilege ambiguity, if classification needs another state, or before permitting a capability snapshot to authorize any mutation or waiver.
